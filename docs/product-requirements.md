# プロダクト要求定義書（product-requirements.md）

## プロダクトビジョンと目的

eigyocare は、社内営業担当者が日々の営業活動の中で必要とする情報（売上・入金状況、取引先マスタ）に、外出先も含めていつでもアクセスできるようにする社内向け営業支援 Web アプリケーションである。

社内基幹システム（SQL Server）に蓄積されているデータを、営業担当者が使いやすい形で参照できるようにすることで、営業活動における情報照会の手間を減らし、顧客対応のスピードと正確性を高めることを目的とする。

## ターゲットユーザーと課題・ニーズ

### ターゲットユーザー
- 社内の小規模営業チーム（数名〜十数名程度を想定）
- 外出中・携帯電話（スマートフォン）からアクセスする営業担当者
- 社内 PC からアクセスする営業担当者・管理部門スタッフ

### 課題・ニーズ
- 売上や入金の状況を確認したいが、基幹システムは外出先から参照しづらい、または営業担当者向けに使いやすい画面になっていない
- 得意先・仕入先の情報をその場ですぐに確認したい（商談中、訪問中など）
- 誰でもアクセスできる状態は避けたい（ログインによる認証が必要）

> **[要確認]** 上記は現時点でのヒアリング内容に基づく仮説。実際の営業担当者が抱える具体的な業務上の困りごと（例：入金遅延の把握が遅れる、訪問前に取引先情報を確認する手段がない等）を追って確認し、必要であれば追記する。

## 主要な機能一覧

本プロダクトは段階的に開発する。**第1目標（現行スコープ）** は全メニューが参照（閲覧）のみの機能であり、**第2目標** で初めて更新系の機能（見積価格設定）を追加する。

### 第1目標（参照機能）

ログイン認証を除き、以下の全メニューは **参照（閲覧）のみ** の機能である（登録・編集は行わない）。

1. **ログイン認証**
   - ユーザー ID／パスワードによるログイン
   - 認証されていないユーザーはデータにアクセスできない
2. **担当者別売上推移** - 担当者別に各月の売上を表示
3. **県別売上推移** - 県別に各月の売上を表示
4. **顧客情報** - 顧客情報を表示
5. **仕入先情報** - 仕入先情報を表示
6. **請求情報** - 請求書の内容を表示
7. **売掛金一覧** - 得意先別に売掛金を表示
8. **入金確認** - 得意先別に入金状況を表示
9. **担当者別売上対比** - 担当者別に月初から昨日までの売上を表示
10. **担当者別予算進捗** - 担当者別に予算に対する売上の進捗を表示
11. **返品** - 返品の件数や金額を表示
12. **値引** - 値引の金額を表示

### 第2目標（更新機能）

13. **見積価格設定** - 事務が基幹システムに登録した見積もりに対して、営業担当者が価格（見積金額）を入力・更新する

> **[要確認]** 第2目標は現時点では方向性の共有段階。以下は `functional-design.md` 作成時に詳細化する。
> - 事務が登録する見積もり情報の範囲（品目・数量など）と、営業が入力・更新する項目（価格のみか、それ以外も含むか）の切り分け
> - 価格設定後、事務側（基幹システム）にどう反映されるか（即時反映か、承認フローを挟むか）
> - 同一見積もりに対する編集権限（担当営業のみか、他の営業も編集可能か）や排他制御（同時編集の防止）
> - 価格変更履歴の記録（監査ログ）の要否

## 成功の定義

> **[要確認]** 定量的な成功指標（KPI）は未定義。例えば以下のような案が考えられるため、方向性を確認したい。
> - 営業担当者が基幹システムへ電話・メールで問い合わせる件数の削減
> - 外出先からの利用率・アクセス頻度
> - 入金遅延の把握・対応スピードの改善

## ビジネス要件

- 社内利用限定（社外への公開は想定しない）
- 既存の社内基幹システム（SQL Server）とのデータ連携が前提
- 最終的に Windows Server 上の IIS で稼働させる
- 営業担当者はスマートフォンからもアクセスする（レスポンシブ対応が必要）
- 第1目標は参照機能のみでリリースし、第2目標として見積価格設定（基幹システムへの更新を伴う機能）を追加する段階的な開発とする

## ユーザーストーリー

- 営業担当者として、外出先からスマートフォンで自分（または他の担当者）の売上推移を確認したい。訪問前・商談中に実績を把握するため。
- 営業担当者として、県別の売上推移を確認したい。エリアごとの状況を把握するため。
- 営業担当者として、顧客情報・仕入先情報をすぐに調べたい。商談中にその場で確認するため。
- 営業担当者として、請求情報・売掛金・入金状況を確認したい。得意先への対応（督促や案内）を判断するため。
- 営業担当者として、自分の予算に対する進捗状況を確認したい。目標達成に向けた行動を調整するため。
- 営業担当者として、返品・値引の状況を確認したい。取引条件や対応状況を把握するため。
- 利用者として、ログインしないと社内データにアクセスできないようにしたい。情報の不正アクセスを防ぐため。
- 営業担当者として（第2目標）、事務が登録した見積もりに対して価格を入力・更新したい。基幹システムの画面を使わずに、外出先からでも見積対応ができるようにするため。

## 受け入れ条件

- ログインしていないユーザーは、いずれのメニューにもアクセスできないこと
- 社内 PC およびスマートフォンのブラウザから、全メニューを参照できること
- 表示されるデータが SQL Server 上の基幹システムのデータと一致していること
- （第2目標）営業担当者が入力・更新した見積価格が、基幹システム側に正しく反映されること

## 機能要件

- ユーザー認証機能（ログイン／ログアウト）
- 担当者別売上推移の参照機能（月次）
- 県別売上推移の参照機能（月次）
- 顧客情報の参照機能
- 仕入先情報の参照機能
- 請求情報の参照機能
- 売掛金一覧の参照機能（得意先別）
- 入金確認の参照機能（得意先別）
- 担当者別売上対比の参照機能（月初〜前日）
- 担当者別予算進捗の参照機能
- 返品情報の参照機能（件数・金額）
- 値引情報の参照機能（金額）
- （第2目標）見積価格の入力・更新機能

> **[要確認]** 各メニューの検索・絞り込み条件、表示項目（カラム）の詳細は `functional-design.md` にて具体化する。

## 非機能要件

- 社内 SQL Server との接続・データ連携ができること
- スマートフォンのブラウザからも問題なく閲覧できること（レスポンシブデザイン）
- Windows Server（IIS）上で稼働できること
- 社内利用者以外がアクセスできないこと（認証・アクセス制御）
- （第2目標）基幹システムへの書き込みが発生するため、データ整合性・同時更新の排他制御を考慮すること

> **[要確認]** パフォーマンス要件（同時アクセス数、レスポンスタイム等）、可用性要件は未定義。必要に応じて追記する。
